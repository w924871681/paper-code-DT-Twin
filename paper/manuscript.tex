\shorttitle{Complexity-Constrained Few-Shot DT Instantiation}

\title[mode=title]{Complexity-Constrained Few-Shot Digital Twin Model Instantiation for Heterogeneous Multi-Center Computing Networks}

\begin{abstract}
Shared edge--cloud digital twin platforms maintain target-specific digital
twin model instances for heterogeneous computing centers to support
continuous monitoring and prediction. Because edge and cloud hosting
locations differ in available computing and memory capacity, each digital
twin model instance operates under upper bounds on the estimated operation
count per inference and the parameter count, thereby defining a
complexity-constrained setting. In this setting, limited target data at a
new or reconfigured center makes reliable architecture selection and
parameter adaptation challenging. Existing methods usually adapt a fixed
architecture or compare several architectures, but they provide limited
support for joint architecture selection and parameter adaptation under
these bounds, and model selection may be unreliable when validation data
are limited. To address
this problem, we propose the Reference-Regularized Complexity-Constrained
Few-Shot Digital Twin Instantiation algorithm (RCF-DTI). RCF-DTI first removes candidates that exceed either limit and then adapts
the remaining source-trained candidates using a common update budget. It
selects an alternative candidate only when its validation loss is
sufficiently lower than that of a fixed reference candidate. Experimental results demonstrate that
RCF-DTI achieves the lowest prediction error among the compared methods,
reducing mean squared error by 14.60\% relative to the
pretraining-and-fine-tuning baseline, while all selected models satisfy both
limits.
\end{abstract}


\begin{highlights}
\item Joint architecture selection and parameter adaptation enable target-specific digital twin model instantiation.
\item Candidates exceeding operation-count or parameter-count limits are removed before adaptation.
\item An alternative is selected only after sufficient improvement over a fixed reference.
\item The analysis establishes feasibility and the minimum gain required for selecting an alternative.
\item RCF-DTI reduces mean squared error by 14.60\% while all selected models satisfy both limits.
\end{highlights}

\section{Introduction}
\label{sec:introduction}

Digital twin platforms maintain virtual model instances of physical systems
to support monitoring, prediction, analysis, and decision making
\cite{ref1,ref2,ref3}. In a heterogeneous multi-center computing network, a
shared edge--cloud platform maintains one target-specific digital twin model
instance for each physical center. Differences in monitored variables,
workload patterns, and data distributions may require different model
architectures. The corresponding model instances may also be hosted at edge
or cloud locations with different computing and memory capacities and
service requirements. Each instance therefore operates under upper bounds
on the estimated operation count per inference and the parameter count,
forming a complexity-constrained setting. In this setting, a new or
reconfigured center may have only limited target data, making reliable
architecture selection and parameter adaptation challenging. Direct reuse
of an existing instance may be unsuitable because it may not fit the target
center or satisfy both bounds. The platform must therefore select a suitable
architecture and adapt its parameters from the available target data while
ensuring that the resulting model satisfies both bounds. We call this joint
process target-specific digital twin model instantiation.
Figure~\ref{fig:scenario} illustrates this setting.

\paperfig[trim={30bp 640bp 200bp 40bp},clip]{0.68\linewidth}
{fig1.pdf}
{Platform-side target-specific digital twin model instantiation in a
heterogeneous multi-center computing network. For a new or reconfigured
target center, only limited target data may be available. An existing model instance may not fit the target center or satisfy the
assigned upper bounds on estimated operation count and parameter count. The shared edge--cloud platform must therefore instantiate a target-specific
digital twin model.}
{fig:scenario}

Existing studies address the main parts of this problem separately. Digital
twin research supports platform-side model management, but it often assumes
a predefined or common model architecture \cite{ref1,ref5,ref7}. Few-shot
and meta-learning methods transfer knowledge from related tasks, but most of
them adapt one fixed architecture \cite{ref9,ref10,ref11}.
Deployment-aware architecture search considers prediction accuracy and
execution cost \cite{ref12,ref13}. Few-shot and zero-shot search reduce
repeated training, but they do not focus on reliable target-specific
selection from limited validation data \cite{ref14,ref15,ref16}. Taken
together, these studies provide limited support for joint architecture
selection and parameter adaptation under target-specific complexity bounds.
Moreover, small validation differences may lead to unreliable model
selection when validation data are limited.

To address this problem, we propose the Reference-Regularized
Complexity-Constrained Few-Shot Digital Twin Instantiation algorithm, termed
RCF-DTI. An offline stage constructs a compact candidate bank of source-trained
models with different architectures. For each target center, RCF-DTI removes
candidates that exceed either complexity limit and adapts the remaining
candidates using the same target data and a common update budget. It selects
the best alternative only when its validation loss is sufficiently lower
than that of a fixed reference candidate. The selected architecture and
adapted parameters form the target-specific digital twin model.
Multi-center workload prediction \cite{ref17,ref18,ref19} is used only for
evaluation.

The main contributions are as follows.

\begin{itemize}

\item The proposed formulation joins architecture selection and parameter
adaptation in one target-specific digital twin instantiation problem with
limited target data and hard operation-count and parameter-count limits. It
includes a fixed reference candidate and requires sufficient validation
improvement before selecting an alternative.

\item RCF-DTI provides an algorithmic solution over a finite candidate bank.
It filters candidates before adaptation because model complexity is fixed
by architecture. It uses a common adaptation budget to make candidate
losses comparable and calibrates the selection threshold before target
instantiation.

\item The analysis establishes that every output satisfies both limits and
that every selected alternative reaches the required validation
improvement. When validation error is bounded, it gives a sufficient
condition for lower expected target loss. It also derives the target-stage
computational complexity.

\end{itemize}

\section{Related Work}
\label{sec:related}

\subsection{Digital Twin Modeling and Platform Deployment}

Digital twins maintain virtual models of physical entities to support
monitoring, prediction, simulation, and control
\cite{ref1,ref2,ref3}. In industrial and networked systems, data-driven
models capture system states and support management decisions
\cite{ref4,ref5,ref6}. Digital twin networks extend this setting to network
elements, computing nodes, and services, whose models may be hosted on cloud
or edge platforms while remaining associated with individual physical
entities \cite{ref7,ref8}. Platform-side studies further consider service
placement and computation offloading under delay and resource constraints
\cite{ref27,ref28}.

These studies establish the basis for platform-side digital twin modeling
and deployment, but they commonly assume a predefined or shared model
architecture. They provide limited support for selecting different
architectures for newly observed heterogeneous centers under
target-specific complexity limits.

\subsection{Few-Shot Learning and Meta-Learning}

Few-shot learning adapts a model to a new task using limited target data,
while meta-learning learns transferable knowledge, initializations, or
update rules from related tasks \cite{ref9,ref10}. Context-aware and cross-domain methods improve transfer under changing
tasks and data conditions \cite{ref20,ref21}. Prompt-based and memory-based
methods provide task-specific information or stored experience
\cite{ref22,ref23}. Related ideas have also been applied to
digital twin construction, online and continual adaptation, and few-shot
workload prediction \cite{ref11,ref24,ref25}.

Despite these advances, most methods adapt parameters within a fixed model
architecture. They provide limited support for jointly selecting an
architecture and adapting its parameters during target-specific digital
twin model instantiation.

\subsection{Deployment-Aware Architecture Selection}

Neural architecture search selects model structures from a predefined
architecture space. Platform-aware and hardware-aware methods consider both
prediction accuracy and execution cost \cite{ref12,ref13}. Few-shot search
shares information among candidate architectures to reduce repeated
training \cite{ref14}, while zero-shot and training-free methods rank
architectures without fully training every candidate
\cite{ref15,ref16}. Multi-objective search further considers several
accuracy and complexity measures during architecture selection
\cite{ref26}.

These methods mainly improve search efficiency, architecture ranking, or
accuracy--complexity trade-offs. They provide limited support for reliable
target-specific model selection under hard complexity limits when target
and validation data are limited.

Overall, existing work mainly treats platform-side digital twin management,
few-shot adaptation, and deployment-aware architecture selection as
separate problems. It provides limited support for joint architecture
selection and parameter adaptation under hard operation-count and
parameter-count limits, especially when validation data are limited.
RCF-DTI addresses these limitations by filtering infeasible candidates,
adapting feasible candidates with a common update budget, and requiring
sufficient improvement over a fixed reference.

\section{System Model and Problem Formulation}
\label{sec:problem}

\subsection{Platform Setting}

We consider a shared edge--cloud digital twin platform that maintains model
instances for heterogeneous computing centers. Let
$\mathcal{C}_{\mathrm{src}}$ denote the source centers that provide
historical data for offline training. During each instantiation process, we consider one new or reconfigured
target center $c$ with limited target data.

Each physical center is associated with one model instance hosted by the
shared platform. The centers may differ in monitored variables, workload
patterns, and data distributions. They may therefore require different
model architectures.

Let $\mathcal{A}$ denote the predefined architecture space. For an
architecture $a\in\mathcal{A}$, let $f_{a,\theta}$ denote a data-driven
digital twin model with parameter vector $\theta$. The digital twin model
instance for target center $c$ is represented by

\begin{equation}
\mathrm{DT}_{c}
=
\left(
a_c,\theta_c
\right),
\label{eq:dt-instance}
\end{equation}

where $a_c$ is the selected model architecture and $\theta_c$ is its
target-adapted parameter vector. Model instantiation must therefore
determine both the architecture and the model parameters.

\subsection{Candidate Models and Complexity Limits}

The target observations are divided in chronological order into a support
set $\mathcal{D}_{c}^{\mathrm{sup}}$ and a validation set
$\mathcal{D}_{c}^{\mathrm{val}}$. The support set is used for target adaptation. The validation set is used
for candidate selection.

A candidate model is denoted by
$q=(a_q,\theta_q^{0})$. Here, $a_q$ is the candidate architecture and
$\theta_q^{0}$ is its source-trained initialization. We call a finite set
of candidate models a candidate bank. Let $\mathcal{Q}$ denote the candidate
bank constructed before target instantiation.
Different candidates may use different architectures or different
source-trained initializations.

The platform assigns two hard complexity limits to the model instance for
target center $c$. These limits may vary with the available capacity at the
hosting location and the service requirements. The first limit controls the
computation required for one inference, and the second controls the model
size. For architecture $a_q$, let $F_c(a_q)$ denote its estimated operation
count for one inference under the input setting of center $c$. Let $P(a_q)$
denote its parameter count. Their assigned upper bounds are denoted by
$B_c^{F}$ and $B_c^{P}$, respectively.

The feasible candidate set is

\begin{equation}
\mathcal{Q}_{c}^{\mathrm{fea}}
=
\left\{
q\in\mathcal{Q}
\;\middle|\;
F_c(a_q)\leq B_c^{F},
\;
P(a_q)\leq B_c^{P}
\right\}.
\label{eq:feasible-candidate-set}
\end{equation}

A candidate belongs to $\mathcal{Q}_{c}^{\mathrm{fea}}$ only when it
satisfies both limits. The two limits are hard constraints. They are not
added to the prediction loss as penalty terms.

One candidate $q^{\mathrm{ref}}$ is fixed as the reference candidate before
target instantiation. We assume that
$q^{\mathrm{ref}}\in\mathcal{Q}_{c}^{\mathrm{fea}}$
for each target center to which the algorithm is applied. The reference candidate provides a fixed default
choice when an alternative candidate does not provide enough validation
improvement.

\subsection{Problem Formulation}

Let $\mathcal{L}(\mathcal{D};a,\theta)$ denote the mean task loss of model
$f_{a,\theta}$ on dataset $\mathcal{D}$. The task loss used in the
experiments is given in Section~\ref{sec:experiments}.

Let $T$ denote the fixed number of target parameter updates. For each
feasible candidate $q$, let $\theta_{c,q}^{(T)}$ denote its parameter
vector after $T$ updates on $\mathcal{D}_{c}^{\mathrm{sup}}$. The concrete
update rule is given in Section~\ref{sec:method}. The post-adaptation
validation loss is

\begin{equation}
J_c(q)
=
\mathcal{L}
\left(
\mathcal{D}_{c}^{\mathrm{val}};
a_q,
\theta_{c,q}^{(T)}
\right).
\label{eq:validation-loss}
\end{equation}

Directly returning the candidate with the lowest value of $J_c(q)$ may be
unreliable when the validation set is small. A small difference on the validation set may disappear on unseen target
data. We therefore add a relative selection
cost when a non-reference candidate is selected. We call this cost
reference regularization.

Let $\tau\in[0,1)$ denote the required relative improvement over the
reference candidate. The regularized selection score is defined as

\begin{equation}
S_c(q;\tau)
=
\begin{cases}
J_c(q),
&
q=q^{\mathrm{ref}},
\\[1mm]
J_c(q)
+
\tau J_c(q^{\mathrm{ref}}),
&
q\neq q^{\mathrm{ref}}.
\end{cases}
\label{eq:regularized-score}
\end{equation}

The added term is used only for candidate selection. It does not change the
parameter adaptation loss. A larger value of $\tau$ requires a larger validation improvement before
the algorithm selects a non-reference candidate.

The target-specific instantiation problem is

\begin{equation}
q_c^{*}
\in
\underset{q\in\mathcal{Q}_{c}^{\mathrm{fea}}}
{\arg\min}
\;
S_c(q;\tau).
\label{eq:instantiation-problem}
\end{equation}

The formulation has two levels. The inner level adapts the parameter vector
of each feasible candidate using the target support set. The outer level
selects one candidate under the complexity limits using
Eq.~\eqref{eq:regularized-score}. We therefore treat target-specific model
instantiation as a constrained two-level decision problem.


\section{Proposed Method}
\label{sec:method}

\subsection{Overview}

RCF-DTI uses a candidate bank prepared offline together with a fixed
reference candidate and a calibrated selection threshold. For each target
center, the algorithm receives the target data and assigned complexity
limits. It first removes candidates that violate either limit, then adapts
the feasible candidates with a fixed update budget, and finally performs
reference-regularized selection. The selected architecture and its adapted
parameters form the target-specific digital twin instance.
Figure~\ref{fig:method-overview} summarizes the overall RCF-DTI workflow
and its relation to digital twin operation. The offline bank preparation
and the target-side stages are detailed in the following subsections.

{fig2.pdf}
{Overall workflow of RCF-DTI and its connection to target-center digital
twin operation.}

\subsection{Offline Candidate Bank Construction}

The candidate-bank input in Figure~\ref{fig:method-overview} is prepared
offline in two steps. First, a compact architecture set is retained from
the predefined space using separate development data. All architectures are
compared using the same adaptation and evaluation procedure. The exact
retention rule is reported in
Section~\ref{subsec:experimental-setup}.

Let
$\mathcal{A}_{\mathrm{bank}}=\{a_m\}_{m=1}^{M}$ denote the retained
architecture set, where $M$ is the number of retained architectures.
Second, let $\mathcal{D}_{\mathrm{src}}$ denote the pooled source-center
training data. For each retained architecture $a_m$, the offline stage
trains one or more source initializations. The $r$-th initialization is trained by

\begin{equation}
\theta_{m,r}^{0}
\in
\underset{\theta}{\arg\min}
\;
\mathcal{L}
\left(
\mathcal{D}_{\mathrm{src}};
a_m,\theta
\right),
\qquad
m=1,\ldots,M,
\quad
r=1,\ldots,R_m,
\label{eq:source-initialization-training}
\end{equation}

where $\theta$ is the parameter variable during source training,
$r$ indexes the stored initializations for architecture $a_m$, and $R_m$
is their total number. Different values of $r$ distinguish different
source-trained initializations obtained for the same architecture. In
practice, the source optimizer gives an approximate solution to
Eq.~\eqref{eq:source-initialization-training}.

Each pair
$q_{m,r}=(a_m,\theta_{m,r}^{0})$ forms one candidate. The complete bank is
$\mathcal{Q}=\{q_{m,r}\}$ for
$m=1,\ldots,M$ and $r=1,\ldots,R_m$. Because each initialization is trained
with its own architecture, it is called an architecture-matched source
initialization.

The reference candidate $q^{\mathrm{ref}}$ is fixed from this bank before
target instantiation and is not changed across target centers.
Figure~\ref{fig:source-bank} details the offline construction of the
candidate bank shown in Figure~\ref{fig:method-overview}.

{fig3.pdf}
{Offline candidate bank construction, including architecture retention,
source-initialization training, and designation of a fixed reference
candidate.}
{fig:source-bank}

\subsection{Candidate Filtering}

For target center $c$, RCF-DTI computes the feasible set using
Eq.~\eqref{eq:feasible-candidate-set}. Filtering compares the estimated
operation count $F_c(a_q)$ and parameter count $P(a_q)$ with their assigned
upper bounds $B_c^{F}$ and $B_c^{P}$, respectively. It does not use the
target support or validation data. Because architecture complexity does not
change during parameter adaptation, a candidate that violates either limit
is removed before target updates. This avoids unnecessary adaptation. The
feasible reference candidate ensures at least one valid output.
Figure~\ref{fig:candidate-filtering} illustrates representative
complexity-filtering outcomes.

\paperfig[trim={240bp 1300bp 40bp 180bp},clip]{0.98\linewidth}
{fig4.pdf}
{Representative candidate-filtering outcomes under the assigned
operation-count and parameter-count limits.}
{fig:candidate-filtering}

\subsection{Target Adaptation}

Each feasible candidate starts from its architecture-matched source
initialization. Let $\eta_t>0$ denote the step size at update $t$. Its
parameters are updated by

\begin{equation}
\begin{aligned}
\theta_{c,q}^{(0)}
&=
\theta_q^{0},
\\
\theta_{c,q}^{(t+1)}
&=
\theta_{c,q}^{(t)}
-
\eta_t
\nabla_{\theta}
\mathcal{L}
\left(
\mathcal{D}_{c}^{\mathrm{sup}};
a_q,
\theta_{c,q}^{(t)}
\right),
\\[-1mm]
&\hspace{42mm}
t=0,\ldots,T-1.
\end{aligned}
\label{eq:target-update}
\end{equation}

Here, $t$ is the update index, $\eta_t$ is the step size, and
$\nabla_{\theta}$ denotes the gradient with respect to the model parameters.
The architecture $a_q$ remains fixed during adaptation. All feasible
candidates use the same support set, task loss, and update budget $T$. These
common conditions make their post-adaptation losses comparable.

After the final update, the algorithm computes $J_c(q)$ using
Eq.~\eqref{eq:validation-loss}. The validation set is used only for
candidate selection. The optimizer and other implementation settings are
reported in Section~\ref{subsec:experimental-setup}.

\subsection{Candidate Selection}

Let
$\mathcal{Q}_{c}^{\mathrm{alt}}
=
\mathcal{Q}_{c}^{\mathrm{fea}}
\setminus\{q^{\mathrm{ref}}\}$
denote the set of feasible alternatives. If this set is empty, the
algorithm returns the reference candidate.

Otherwise, let $\widetilde q_c$ denote a feasible alternative with the
lowest post-adaptation validation loss:

\[
\widetilde q_c
\in
\underset{q\in\mathcal{Q}_{c}^{\mathrm{alt}}}
{\arg\min}
\;
J_c(q).
\]

The final candidate is selected by

\begin{equation}
q_c^{*}
=
\begin{cases}
\widetilde q_c,
&
J_c(\widetilde q_c)
\leq
(1-\tau)J_c(q^{\mathrm{ref}}),
\\[1mm]
q^{\mathrm{ref}},
&
\text{otherwise}.
\end{cases}
\label{eq:reference-regularized-selection}
\end{equation}

Equation~\eqref{eq:reference-regularized-selection} returns a minimizer of
the score-based problem in Eq.~\eqref{eq:instantiation-problem} because
every alternative receives the same added selection cost.

Figure~\ref{fig:adaptation-selection} illustrates the fixed-budget
adaptation and reference-regularized selection stages.

\paperfig[trim={100bp 600bp 200bp 140bp},clip]{0.96\linewidth}
{fig5.pdf}
{Fixed-budget target adaptation and reference-regularized selection.
The best feasible alternative is selected when
$J_c(\widetilde q_c)\leq(1-\tau)J_c(q^{\mathrm{ref}})$; otherwise, the
fixed reference candidate is retained.}
{fig:adaptation-selection}

Algorithm~\ref{alg:rcf-dti} summarizes the target-side execution of RCF-DTI
for one target center.

\begin{algorithm}[t]
\caption{RCF-DTI for Target-Specific Model Instantiation}
\label{alg:rcf-dti}
\begin{algorithmic}[1]
\Require Candidate bank $\mathcal{Q}$, reference candidate
$q^{\mathrm{ref}}$, support set $\mathcal{D}_{c}^{\mathrm{sup}}$, and
validation set $\mathcal{D}_{c}^{\mathrm{val}}$
\Require Limits $B_c^{F}$ and $B_c^{P}$, update budget $T$, step sizes
$\{\eta_t\}_{t=0}^{T-1}$, and calibrated threshold $\tau$
\Ensure Target-specific digital twin model instance
$\mathrm{DT}_{c}^{*}$

\State Compute $\mathcal{Q}_{c}^{\mathrm{fea}}$ using
Eq.~\eqref{eq:feasible-candidate-set}

\ForAll{$q\in\mathcal{Q}_{c}^{\mathrm{fea}}$}
    \State $\theta_{c,q}^{(0)}\gets\theta_q^{0}$
    \For{$t=0$ to $T-1$}
        \State
        $g_t\gets
        \nabla_{\theta}
        \mathcal{L}
        \left(
        \mathcal{D}_{c}^{\mathrm{sup}};
        a_q,\theta_{c,q}^{(t)}
        \right)$
        \State
        $\theta_{c,q}^{(t+1)}
        \gets
        \theta_{c,q}^{(t)}-\eta_t g_t$
    \EndFor
    \State Compute $J_c(q)$ using Eq.~\eqref{eq:validation-loss}
\EndFor

\State
$\mathcal{Q}_{c}^{\mathrm{alt}}
\gets
\mathcal{Q}_{c}^{\mathrm{fea}}\setminus\{q^{\mathrm{ref}}\}$

\If{$\mathcal{Q}_{c}^{\mathrm{alt}}=\emptyset$}
    \State $q_c^{*}\gets q^{\mathrm{ref}}$
\Else
    \State Choose
    $\widetilde q_c\in
    \underset{q\in\mathcal{Q}_{c}^{\mathrm{alt}}}{\arg\min}\;J_c(q)$
    \If{$J_c(\widetilde q_c)
    \leq(1-\tau)J_c(q^{\mathrm{ref}})$}
        \State $q_c^{*}\gets\widetilde q_c$
    \Else
        \State $q_c^{*}\gets q^{\mathrm{ref}}$
    \EndIf
\EndIf

\State
$\mathrm{DT}_{c}^{*}
\gets
\left(
a_{q_c^{*}},
\theta_{c,q_c^{*}}^{(T)}
\right)$

\State \Return $\mathrm{DT}_{c}^{*}$
\end{algorithmic}
\end{algorithm}

\subsection{Threshold Calibration}

The threshold $\tau$ is fixed before target instantiation. It is selected
from a finite set $\mathcal{T}$ using a separate set of development cases,
denoted by $\mathcal{C}_{\mathrm{dev}}$. Each development case contains support, validation, and check data. The
support and validation data are used during candidate adaptation and
selection. The check data are not used for within-case adaptation or
candidate selection. They are used only in offline development procedures,
including architecture retention and threshold calibration.

For each case $d\in\mathcal{C}_{\mathrm{dev}}$, let $h_d(\tau)=1$ when
RCF-DTI selects a non-reference candidate whose check loss is higher than
that of the reference candidate. Otherwise, let $h_d(\tau)=0$. The empirical
harmful-selection rate is

\begin{equation}
\widehat R_{\mathrm{harm}}(\tau)
=
\frac{1}{|\mathcal{C}_{\mathrm{dev}}|}
\sum_{d\in\mathcal{C}_{\mathrm{dev}}}
h_d(\tau).
\label{eq:harmful-selection-rate}
\end{equation}

Calibration also requires positive development gains. Let
$\widehat G_{\mathrm{ref}}(\tau)$ denote the mean relative check-loss
reduction over the fixed reference candidate. To distinguish gains from
architecture selection from gains caused only by a different source
initialization, we define a reference-architecture control. This control
applies the same selection rule only to candidates that use the reference
architecture. Let $\widehat G_{\mathrm{arch}}(\tau)$ denote the mean
additional reduction over this control.

Let $\underline G_{\mathrm{ref}}(\tau)$ and
$\underline G_{\mathrm{arch}}(\tau)$ denote the lower confidence bounds of
the two gains. Let $\rho$ denote the allowed harmful-selection rate, and let
$\gamma_{\mathrm{ref}}$ and $\gamma_{\mathrm{arch}}$ denote the required
minimum gains. The eligible threshold set and the calibrated threshold are

\begin{equation}
\begin{aligned}
\mathcal{T}_{\mathrm{elig}}
=
\bigl\{
\tau\in\mathcal{T}
\;\bigm|\;&
\widehat R_{\mathrm{harm}}(\tau)\leq\rho,
\\
&
\widehat G_{\mathrm{ref}}(\tau)\geq\gamma_{\mathrm{ref}},
\quad
\underline G_{\mathrm{ref}}(\tau)>0,
\\
&
\widehat G_{\mathrm{arch}}(\tau)\geq\gamma_{\mathrm{arch}},
\quad
\underline G_{\mathrm{arch}}(\tau)>0
\bigr\},
\\
\tau^{*}
&=
\min \mathcal{T}_{\mathrm{elig}}.
\end{aligned}
\label{eq:threshold-calibration}
\end{equation}

We assume that $\mathcal{T}_{\mathrm{elig}}$ is nonempty. The smallest
eligible threshold is used because it satisfies the development criteria
without requiring a larger validation improvement than necessary. The tested values, gain requirements, and confidence procedure are reported in
Section~\ref{subsec:experimental-setup}.

This calibration controls the observed behavior on the development cases.
It does not guarantee improvement for every unseen target center.

\subsection{Properties and Computational Complexity}

\textbf{Complexity feasibility.}
Algorithm~\ref{alg:rcf-dti} selects a candidate only from
$\mathcal{Q}_{c}^{\mathrm{fea}}$. Therefore, its output always satisfies

\[
F_c(a_{q_c^{*}})\leq B_c^{F},
\qquad
P(a_{q_c^{*}})\leq B_c^{P}.
\]

This property follows directly from the hard filtering step.

\textbf{Minimum validation improvement.}
Suppose that the algorithm returns a non-reference candidate and that
$J_c(q^{\mathrm{ref}})>0$. Equation~\eqref{eq:reference-regularized-selection}
gives

\[
\frac{
J_c(q^{\mathrm{ref}})
-
J_c(q_c^{*})
}{
J_c(q^{\mathrm{ref}})
}
\geq
\tau.
\]

Thus, every accepted alternative reaches at least the required relative
validation improvement.

\textbf{Selection under bounded validation error.}
Let $R_c(q)$ denote the expected loss of the adapted candidate $q$ on future
observations from target center $c$. Suppose that the validation loss
approximates this expected loss with error at most $\epsilon_c$ for the
selected alternative and the reference candidate:

\[
\left|R_c(q)-J_c(q)\right|
\leq
\epsilon_c,
\qquad
q\in\{q_c^{*},q^{\mathrm{ref}}\}.
\]

If RCF-DTI selects a non-reference candidate, then
Eq.~\eqref{eq:reference-regularized-selection} gives

\begin{equation}
R_c(q_c^{*})
-
R_c(q^{\mathrm{ref}})
\leq
-\tau J_c(q^{\mathrm{ref}})
+
2\epsilon_c.
\label{eq:bounded-validation-error}
\end{equation}

Therefore, when
$\tau J_c(q^{\mathrm{ref}})>2\epsilon_c$, the selected alternative has a
lower expected loss than the reference candidate on future target
observations. This is a sufficient condition under the stated validation
error bound. It does not provide an unconditional guarantee for every
target center.

\textbf{Target-stage computation.}
Let $C_q^{\mathrm{grad}}$ denote the cost of one gradient update for
candidate $q$, and let $C_q^{\mathrm{val}}$ denote its validation cost. The
target-stage computation is

\begin{equation}
O
\left(
|\mathcal{Q}|
+
\sum_{q\in\mathcal{Q}_{c}^{\mathrm{fea}}}
\left(
T C_q^{\mathrm{grad}}
+
C_q^{\mathrm{val}}
\right)
\right).
\label{eq:target-complexity}
\end{equation}

Here, $O(\cdot)$ denotes asymptotic computational complexity. Given the
complexity values available at filtering time, the first term is the
filtering cost. The remaining terms are the adaptation and validation costs
of the feasible candidates. Filtering before adaptation avoids the
update cost of candidates that cannot satisfy the platform-assigned limits.

The algorithm returns $\mathrm{DT}_{c}^{*}$ and records
$F_c(a_{q_c^{*}})$ and $P(a_{q_c^{*}})$ for the selected architecture.

\section{Experimental Evaluation}
\label{sec:experiments}

This section evaluates average prediction error, performance on difficult
cases, limit satisfaction, target-side cost, and generalization. It also
examines the main components and configuration choices of RCF-DTI.

\subsection{Experimental Setup}
\label{subsec:experimental-setup}

\subsubsection{Benchmark and Evaluation Protocol}

We use a controlled synthetic benchmark for heterogeneous multi-center
workloads. The benchmark varies sampling intervals, missing-data patterns,
noise levels, workload dynamics, target data sizes, and model complexity
limits. Each center contains 3000 workload observations. The input window
length is $L=96$.

The main evaluation uses 20 source centers and 20 held-out target centers.
For each target center, we consider two prediction horizons,
$H\in\{1,4\}$, and two support sizes, $K\in\{10,20\}$. A target case is one
combination of a target center, a prediction horizon, and a support size.
The main evaluation therefore contains 80 target cases.

The target centers are grouped into types A, B, and C. These groups differ
in sampling intervals, missing-data patterns, noise levels, and workload
dynamics. The group label is used only for result analysis. It is not an
input to RCF-DTI or any baseline.

We use four disjoint center groups: source, development, diagnostic, and
held-out target. Source centers train the source initializations.
Development centers are used for architecture retention and threshold
calibration. The diagnostic group contains three disjoint pools for the
ablation, optimizer-matched, and source-scale studies. Held-out target
centers are used only for the main comparison.

Each target case is divided in chronological order into support, validation,
check, and test sets. Support data are used for adaptation, and validation
data are used for candidate selection. On development cases, validation and
check data support architecture retention, and check data also calibrate the
threshold. On diagnostic and held-out cases, check data are used only after
candidate selection. Test data are used only for final evaluation. Neither
check nor test data affect target adaptation or within-case candidate
selection.

The reference architecture is a one-layer gated recurrent unit (GRU) with
32 hidden units and a dropout rate of 0.1. Its parameters are pretrained on
the pooled source-center data. The resulting source-trained model is used as
the reference candidate in all later experiments.

An alternative architecture is retained if, in at least two development
cases, its error is lower than that of the reference architecture on the
validation set or the check set. This rule fixes the retained architecture set before any diagnostic or
held-out evaluation.

The selection threshold is fixed on the development cases. A harmful
selection occurs when RCF-DTI selects a non-reference candidate whose check
loss is higher than that of the reference candidate. We test thresholds of
5\%, 7.5\%, 10\%, 12.5\%, 15\%, and 20\%.

A threshold is eligible only when all development criteria are satisfied.
First, its harmful-selection rate must not exceed 5\%. Second, its mean
check-loss reduction over the reference candidate must be at least 3\%,
and the lower bound of its 95\% confidence interval must be positive.
The additional reduction over the reference-architecture control must
satisfy the same gain and confidence requirements. The smallest eligible threshold is 10\%. The main candidate bank, reference
candidate, and selection threshold are then fixed. They are not changed
using the ablation, optimizer-matched, or held-out results.

\subsubsection{Task, Metrics, and Implementation}

We use workload prediction only as the evaluation task. It tests whether an
instantiated digital twin model captures the temporal behavior of the target
center.

Let $\mathbf{X}_{c,t}$ denote an input window of length $L$. The model
predicts the next $H$ workload states as

\begin{equation}
\widehat{\mathbf{y}}_{c,t+1:t+H}
=
f_{a,\theta}
\left(
\mathbf{X}_{c,t}
\right),
\label{eq:prediction-task}
\end{equation}

where $t$ is the current time index,
$\widehat{\mathbf{y}}_{c,t+1:t+H}$ is the predicted workload vector, and
$\mathbf{y}_{c,t+1:t+H}$ is the corresponding observed vector.

We use mean squared error (MSE) as the task loss. For a dataset
$\mathcal{D}$ of input--output pairs, it is defined as

\begin{equation}
\mathcal{L}
\left(
\mathcal{D};a,\theta
\right)
=
\frac{1}{|\mathcal{D}|}
\sum_{(\mathbf{X},\mathbf{y})\in\mathcal{D}}
\frac{1}{H}
\left\|
f_{a,\theta}(\mathbf{X})-\mathbf{y}
\right\|_{2}^{2}.
\label{eq:prediction-loss}
\end{equation}

Here, $|\mathcal{D}|$ is the number of prediction windows, $H$ is the
prediction horizon, and $\|\cdot\|_2$ is the Euclidean norm. The loss
averages the squared error over all prediction windows and prediction
steps. It is the task loss introduced in Section~\ref{sec:problem}.

We report mean absolute error (MAE), MSE, Worst-10\% error, conditional value
at risk at the 90th percentile (CVaR90), target-side instantiation time, and
limit satisfaction. For each target case, MAE and MSE are averaged over all
test windows and prediction steps. Worst-10\% error is the mean window-level
MSE over the 10\% of test windows with the largest errors. CVaR90 is the
mean case-level MSE over the 10\% of target cases with the largest MSE
values. Limit satisfaction is the percentage of selected models that meet
both assigned model complexity limits.

The main results report averages over the held-out target cases. Since all
methods are evaluated on the same target cases, relative gains are first
computed for each paired case. The four $(H,K)$ cases from the same target center are treated as one group.
The 95\% confidence intervals are obtained by resampling target centers
4000 times.

All methods are implemented in Python and PyTorch and run on the same NVIDIA
RTX 3060 Laptop GPU. The main source initializations are trained with the
Adam optimizer for 50 epochs. The learning rate is $10^{-3}$, and the batch
size is 64.

Each target update uses the complete support set. The updates are implemented
with the SGD optimizer and MSE loss. All feasible RCF-DTI candidates use the
same learning rate, gradient clipping rule, and fixed budget of 50 updates.
Candidate-specific early stopping is not used.

GPU execution is synchronized before runtime measurement. The timer includes candidate model setup, loading of source initializations,
target adaptation, validation evaluation, and final candidate selection. It
excludes source training, data generation, and post-selection check and test
evaluation.

The main candidate bank uses source-training seed 2904. Seeds 2905 and 2906
are used only in the additional training-seed study.


\subsubsection{Baselines and Comparison Settings}

We compare RCF-DTI with six baselines: pretraining and fine-tuning (PT+FT),
MeDeT-based adaptation, random initialization, few-shot neural architecture
search, zero-shot neural architecture search, and zero-shot neural
architecture search with fine-tuning.

PT+FT adapts the source-trained reference candidate using the same SGD, MSE,
and 50-step protocol as RCF-DTI. Random initialization uses the same
reference architecture and adaptation protocol without source training.
MeDeT-based adaptation uses source parameters learned with
Model-Agnostic Meta-Learning (MAML) for the fixed reference architecture,
following the few-shot adaptation setting of MeDeT~\cite{ref11}.
Few-shot neural architecture search is an adapted comparison based on the
candidate sharing and ranking setting in~\cite{ref14}. It retains and adapts
the top 12 candidates. The two zero-shot
baselines use architecture scores that do not require full target training
\cite{ref15,ref16}. One selects a model without target adaptation. The other
adapts its top 12 candidates before final selection.

The MeDeT and architecture-search baselines are adapted to the common
workload prediction setting. They are not full reproductions of the original
systems. All methods use the same target cases, chronological splits,
complexity limits, and architecture definitions. No method uses the target
test set for training, selection, or stopping.

PT+FT is the main controlled baseline because it adapts the same reference
candidate as RCF-DTI. We also conduct an optimizer-matched comparison on a
separate diagnostic pool. In this comparison, all adapted candidates use
SGD, MSE, and 50 updates, and each matched search baseline evaluates 12
candidates. The pool is not used to tune any method.

\input{tables/table5_ablation.tex}

We first examine the source-training procedure and the role of architecture
selection. The pooled-source variant keeps the candidate architectures and

the assigned hard limits rather than directly reducing prediction error.

\subsection{Generalization Study}
\label{subsec:generalization}

We examine RCF-DTI under changes in source-center scale, source-training
seed, and workload data. These studies use the same target-side adaptation
and selection procedure as the main evaluation. In each study, RCF-DTI is
compared with a reference candidate trained under the same source condition
and source-training budget. We call this candidate the matched reference.

\textbf{Source-center scale.}
We rebuild the candidate bank using 10, 20, 30, 40, and 50 source centers.
Each source-trained model receives 2000 source-training updates at every
scale. At each scale, RCF-DTI and the matched reference use the same
source-center set and source-training budget. The evaluation uses the
separate source-scale diagnostic target pool.

Figure~\ref{fig:generalization}(a) reports the paired MSE reduction relative
to the matched reference at each source scale. The
20-source-center result belongs to this diagnostic study. It is therefore
not directly comparable with the 14.60\% main result, which is obtained on
the held-out target centers.

The mean reduction is positive at all five source scales, but it does not
increase monotonically. Under a fixed source-training budget, adding more
source centers distributes the same number of updates across more source
workloads. More source centers therefore do not always produce a larger
target-side gain.

\textbf{Source-training seeds and external data.}
We rebuild the candidate bank using source-training seeds 2904, 2905, and
2906. The three banks use the same source centers, architecture set,
source-training budget, and data-generation seed. They are evaluated on one fixed diagnostic target pool with identical
target adaptation seeds and the same target-side protocol. The only changed factor is the source-training
seed. This study therefore examines the sensitivity of the candidate bank
to source-training randomness.

We also apply the fixed RCF-DTI configuration to machine-usage data from
Alibaba Cluster Trace v2018~\cite{ref29}. The workload observations are
real. The 20 target machines are assigned deterministically to the tight,
medium, and loose complexity settings in groups of 6, 8, and 6,
respectively, using a fixed hash-based ordering of machine identifiers.
These settings use the same operation-count and parameter-count bounds as
the synthetic benchmark. The candidate bank, adaptation procedure,
filtering rule, and selection threshold are not retuned.

\paperfig{0.98\linewidth}
{fig12.pdf}
{Generalization under changed source conditions.
(a) Paired MSE reduction relative to the reference trained at each source
scale; shading shows 95\% confidence intervals.
(b) Case-level reductions for three source-training seeds and Alibaba
v2018. Four Alibaba cases below $-25\%$ are clipped only for display; all

Across the three source-training seeds, the mean paired MSE reduction ranges

retuning, but it does not show a consistent improvement on this dataset.

\section{Conclusion}
\label{sec:conclusion}

This paper studied complexity-constrained few-shot digital twin model
instantiation for heterogeneous multi-center computing networks. We
formulated target-specific instantiation as joint architecture selection
and parameter adaptation under limits on the estimated operation count per
inference and the parameter count. We then proposed RCF-DTI as an
algorithmic solution. RCF-DTI filters infeasible candidates, adapts the
feasible candidates with a common update budget, and retains the fixed
reference unless an alternative provides sufficient validation
improvement. Experiments show that RCF-DTI achieves lower prediction error
than all compared methods while every selected model satisfies both limits.
The mean gains remain positive across heterogeneous target settings and
source-training seeds, whereas the confidence interval on Alibaba v2018
includes zero. Future work will evaluate actual latency, memory use, and
energy consumption on deployed edge--cloud platforms.
